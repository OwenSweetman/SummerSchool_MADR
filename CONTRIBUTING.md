# Contributing to the ADR Manager VS Code Extension

Thanks for your interest in contributing. This document covers everything you need to get a local development environment running, understand the repository layout, and submit a change.

## Code of conduct

This project follows the [Contributor Covenant](CODE_OF_CONDUCT.md). By participating, you agree to uphold its terms.

## Getting started

### Prerequisites

* [Node.js](https://nodejs.org/) 14.x or later (the `engines.vscode` field targets VS Code 1.67+)
* [Visual Studio Code](https://code.visualstudio.com/) 1.67+ for running the extension locally
* Java 8+ (only needed if you intend to regenerate the ANTLR parser — see [below](#regenerating-the-parser))

### Local setup

```bash
git clone https://github.com/OwenSweetman/SummerSchool_MADR.git
cd SummerSchool_MADR
npm install
```

### Running the extension

1. Open the repository in VS Code.
2. Press `F5` (or run the `Run Extension` launch configuration). A new VS Code window opens with the extension loaded.
3. Open a folder containing an `ADR Directory` (default `docs/decisions`) and try the `Open ADR Manager` command from the Command Palette.

### Running the tests

```bash
npx jest                         # full test suite (~65 tests)
npx jest src/test/parser.test.ts # parser + TC annotation tests
npm run compile-tests            # TypeScript type-check only
npm run compile                  # webpack build of the extension runtime
```

All tests must pass before opening a PR.

## Workflow

### Branching

* `master` is the integration branch — all merged work lands here.
* Feature work happens on a topic branch off `master`. Convention used during the summer school programme: `track-a-*`, `track-b-*`, etc. For external contributors, any descriptive branch name is fine.

### Before pushing

```bash
npx jest                # all tests pass
npm run compile-tests   # TypeScript clean
npm run compile         # webpack clean
```

### Commits

Use clear, imperative commit messages (`Add foo`, `Fix bar`, `Refactor baz`). Squash trivial fix-up commits before opening a PR.

### Pull requests

Use the [PR template](.github/pull_request_template.md). Required:

* Description of the change
* Test plan (what you ran, what you verified)
* Linked issue if applicable

All PRs are run through [CI](.github/workflows/ci.yml) — tests + compile must be green before merge.

## Regenerating the parser

The parser at `src/plugins/parser/` is generated from `src/plugins/parser/MADR.g4` using ANTLR. **Never hand-edit the generated `.js` files** — they will be overwritten on the next regeneration.

To regenerate (after editing `MADR.g4`):

```bash
# One-time setup: install ANTLR tooling
pip install --user antlr4-tools

# Regenerate
cd src/plugins/parser
antlr4 -v 4.9.2 -Dlanguage=JavaScript -o . MADR.g4
```

This produces `MADRLexer.js`, `MADRParser.js`, and `MADRListener.js`. The `.interp` and `.tokens` files ANTLR also produces are gitignored — they're regenerable tooling metadata used by ANTLR IDE plugins, not by the runtime.

After regeneration, you'll typically need to update `src/plugins/parser.js` (the `MADRGenerator` listener) to handle any new rules you added.

## Adding a MADR field

The high-level path for adding a new field to the MADR schema (e.g. a new optional section) touches the following layers in order:

1. **Grammar** — add the rule and token in `src/plugins/parser/MADR.g4`, then regenerate
2. **Data model** — add the field to `ArchitecturalDecisionRecord` in `src/plugins/classes.ts` (declaration, constructor default, `cleanUp()`, `update()`)
3. **Listener** — populate the new field in `src/plugins/parser.js` (`enterX` method)
4. **Serializer** — emit the field in `adr2md` in `src/plugins/parser.js`
5. **Linting** — if the field has format requirements, add a diagnostic in `src/diagnostics/`
6. **Templates** — update `src/plugins/constants.ts` and `snippets/snippets.json`
7. **Webview** — add a form component under `web/` so the field is editable in the UI
8. **Tests** — add fixtures to `src/test/constants.ts` and assertions to `src/test/parser.test.ts`

## Code style

* TypeScript: strict where possible; the project's `tsconfig.json` is authoritative
* JavaScript: only `src/plugins/parser.js` is `.js` (legacy reason — pending TS migration). Other source should be TypeScript
* Indentation: tabs in TS/JS files (matches existing style)
* Linting: `npm run lint` runs ESLint over `src/**/*.ts`. Fix all errors before opening a PR

* What actually happened
